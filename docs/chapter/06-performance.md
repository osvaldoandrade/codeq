# Performance

codeQ's performance story is not "we go fast." It is "we identified specific mutex bottlenecks, fixed them, and measured what came out the other side." Each number in this chapter reflects a tuning intervention against a specific contention point — a `http.Transport.tryPutIdleConn` mutex on the client, a `pebble.commitPipeline` mutex on the storage path, a per-Apply round-trip on the raft path. None of these numbers came from headline-chasing; they came from `go tool pprof -top -cum mutex.pb.gz` runs that pointed at one symbol, followed by a change in that symbol's call site, followed by another bench. Read everything below with that framing in mind.

This chapter covers four things. It establishes the single-node throughput ceiling on a single Pebble shard with gRPC streams, and contrasts it with the HTTP REST baseline so the cost of client-side connection pooling is visible. It walks through what HA costs when raft is enabled, what the Apply coalescer recovers, and why the loopback bench understates real-hardware numbers. It explains the stepwise scaling decision — single shard, multi-shard, 3-node raft, multi-shard raft — and where each lands on the throughput / availability curve. It closes with the bench harness itself: the files that produce the numbers above, what they actually measure, and the honest variance you will see when you run them on your own machine.

## Measured single-node throughput

codeQ sustains 76,639 tasks/s on a single Pebble shard with gRPC streams under the bench harness in `internal/bench/profile_full_cycle_test.go`. The bench boots an in-process Application with the producer-stream and worker-stream endpoints both wired through gRPC, then drives a closed-loop workload: 32 producer goroutines pump task creates over a single persistent producer Session (`internal/bench/profile_full_cycle_test.go:54-70`), while a worker client with 128 concurrent claim slots drains the queue (`internal/bench/profile_full_cycle_test.go:76-87`). Each "task" counted toward the throughput number is a full create + claim + submit cycle, not a single API call. After a 2-second warmup the bench measures for a fixed window and divides ops by elapsed wall time.

| Topology | Transport | Throughput | Source |
|---|---|---|---|
| 1 node, 1 Pebble shard | gRPC streams, NoSync | 76,639 tasks/s | `internal/bench/profile_full_cycle_test.go` |
| 3 nodes, 1 shard, raft | HTTP smart routing | ~3,949 cycles/s | `pkg/app/raft_smart_routing_bench_test.go` |
| 3 nodes, 4 shards, raft | HTTP smart routing | ~3,883 cycles/s | `pkg/app/raft_smart_routing_bench_test.go` |
| 3 nodes, 1 shard, raft | gRPC streams + Apply coalescer | ~9-10K cycles/s | `pkg/app/raft_grpc_bench_test.go` |

The gap between 76,639 tasks/s on the single-node gRPC path and the ~3,900 cycles/s reported by the HTTP smart-routing bench is almost entirely client-side. Phase 0 mutex profiling identified `http.Transport.tryPutIdleConn` as the contended symbol — 28.74% of total mutex time at saturation, which is called out inline in the bench header comment at `pkg/app/raft_grpc_bench_test.go:25-30`. Every HTTP request released its connection back into the client's idle pool through a single mutex, and at a few thousand requests per second on one client process that mutex serialised everything. The fix wasn't to make HTTP faster; it was to stop using a connection pool per request. gRPC streams hold one persistent HTTP/2 connection per Session and multiplex frames over it, so concurrent calls share the connection at the protocol layer instead of through a process-wide pool. The server didn't change. The transport did.

Be honest about what 76,639 tasks/s measures. It is a 12-core box with everything in one process, NoSync mode (`internal/repository/pebble/db.go:135-138`), a single Pebble shard, no replication, no raft, gRPC over loopback. Turn on `FsyncOnCommit=true` and every commit waits for the disk to acknowledge an fsync — the bench-time advantage of the group-commit coalescer (`internal/repository/pebble/db.go:71-82`, with `maxMergeBatch=64` at line 122) shrinks because the bottleneck moves from mutex contention to disk durability. The 76k number is the ceiling for a configuration that explicitly trades durability for throughput, and it should be read with those caveats attached.

The commit coalescer is the reason the single-node number lands where it does. Pebble's internal `commitPipeline` acquires a global mutex on every `Commit()`, and Phase 0 profiles pinned that mutex at 96% of total mutex time and 44% of total block time at 26K req/s (`internal/repository/pebble/db.go:71-82`). The fix was a coalescer goroutine in front of the Pebble DB that merges up to 64 concurrently-arriving batches into a single Pebble batch and calls `Commit()` once. Sixty-four lock acquisitions collapse into one — the primary lever between a measured 26K req/s and the 76K above it.

## The cost of HA

Turn raft on, throughput drops to roughly 10K cycles/s on the same machine. The drop is not subtle: 76,639 tasks/s becomes ~9-10K cycles/s in `pkg/app/raft_grpc_bench_test.go`, an eight-fold reduction. The cause is not raft being slow in absolute terms — it is that the bench runs three codeq processes on one host, on loopback, sharing one CPU and one disk.

Every write in raft mode pays a network-equivalent round-trip for the AppendEntries call that ships the log entry to followers, a majority-ack wait for at least one follower to fsync the log, and an FSM Apply on every replica that reapplies the batch through Pebble. On three real hosts with separate disks and NICs, those costs run in parallel on independent hardware. On loopback inside one machine, they serialise on the same kernel network stack and the same disk queue, and the three FSM Apply calls compete for the same Pebble commit pipeline. The loopback bench is closer to a worst case than a realistic deployment.

What the Apply coalescer recovers is the per-Apply overhead. Each `raft.Apply` call pays a fixed cost regardless of payload size: log append, AE round-trip, majority-ack, FSM Apply. Coalescing N small applies into one larger merged batch amortises that fixed cost across N user operations. The bench shows the baseline (no coalescer) sits at 7-8K cycles/s and the coalesced path sits at 9-10K — a 30-50% lift, measured against the same harness on the same machine in the same run (`pkg/app/raft_grpc_bench_test.go`). The cap is `raftMergeBatch=128` (`internal/raft/db.go:142-149`); at 256 the merged batch starts to exceed raft's preferred entry size and the throughput drops back to where it started.

The open question is what this same workload looks like on three real hosts. The honest answer is we have not measured. Based on the loopback contention pattern — three processes fighting for one disk and one NIC — separating onto independent hardware should remove most of the inter-process contention and recover something in the 2-4× range over the loopback number. That would put 3-node raft somewhere around 20-40K cycles/s on real boxes. But that is a model, not a measurement, and the chapter does not claim a number it did not measure. Run the bench on your own three hosts and the answer is yours, not ours. WSL2 loopback variance compounds this: we have seen the same bench peak at ~20K cycles/s under cache-warm conditions and settle back to 9K after sustained runs, with no configuration change in between.

## When and how to scale

Scaling decisions in codeQ are stepwise, and most workloads land on the first step. A single Pebble shard with gRPC streams sustains 76,639 tasks/s on a 12-core box (`internal/bench/profile_full_cycle_test.go`). For the vast majority of deployments that is the whole story — provision one node, run it, monitor the commit-pipeline mutex time in the profile, and revisit only when the profile shows contention you cannot tune away with the existing knobs in `docs/17-performance-tuning.md`.

If the profile shows commit-pipeline contention as the dominant mutex symbol — and the workload still wants more throughput on one host — the next step is multi-shard on the same node. Each shard runs its own Pebble instance with its own commit-pipeline mutex and its own coalescer goroutine, so doubling or quadrupling the shard count multiplies the number of independent commit pipelines. The smart-routing bench measured 1 shard at ~3,949 cycles/s and 4 shards at ~3,883 cycles/s on the 3-node HTTP path (`pkg/app/raft_smart_routing_bench_test.go`), which is a wash — but that bench is HTTP-bottlenecked, not commit-bottlenecked, so it cannot show the multi-shard win. The empirical sweet spot on the gRPC path is around 4 shards; beyond that the per-shard amortisation gets diluted by inter-shard coordination on shared resources.

If HA is required, you accept the throughput drop and run 3-node raft. The loopback bench measures ~9-10K cycles/s with the Apply coalescer enabled (`pkg/app/raft_grpc_bench_test.go`), which is the realistic ceiling on one machine. On real hosts the number is higher; we have not measured how much higher.

If you need HA and parallelism, run 3-node raft with 4 Pebble shards — the M2 deployment mode. Each shard becomes an independent raft group with its own leader, and leadership distributes across the three nodes so writes for different shards land on different leaders. The decision table for picking among these four configurations lives in `docs/41-deployment-modes.md`; the historical numbers for each live in `docs/30-performance-baselines.md`.

The one mode this chapter deliberately does not promote is the Phase 5 consistent-hash cluster path. It is preserved in the codebase for reference and for users on existing deployments, but the recommended HA path is 3-node raft. The decision rationale, the trade-off, and the migration story are covered in `docs/41-deployment-modes.md` and `docs/40-raft-replication.md`.

## What the bench harness measures

The bench files in this repository are the source of truth for every number in this chapter. Each test sets up an in-process Application — the same `pkg/app.Application` you would run in production, instantiated with the same config plumbing — drives synthetic load through the real gRPC and repository code paths, and reports throughput. There is no mock anywhere in the hot path. The numbers reflect what the real binary does under bench-shaped load, not what an idealised model would do.

`internal/bench/profile_full_cycle_test.go` is the single-node harness. It boots the Application with a producer-stream port and a worker-stream port, opens 32 producer goroutines feeding a single producer Session, and drives a worker client with 128 concurrent claim slots. It runs for a fixed window after warmup and reports producer-side and worker-side rates separately. If the rates do not match, the queue is filling: the worker is not draining as fast as the producer is filling. Both rates are interesting; a 76,639 tasks/s number means producer-side throughput, with worker-side keeping pace. The same harness also writes CPU, alloc, block, and mutex pprof profiles to `/tmp/codeq-profiles` for the run that produced the number — every claim in this chapter is one `go tool pprof` invocation away from being checked.

`pkg/app/raft_grpc_bench_test.go` is the multi-node harness. It uses `startGRPCCluster` to bring up three codeq processes in-process with raft replication enabled, opens four producer Sessions per node (because gRPC `stream.Send` serialises per-stream and we want concurrent submitters per node), wires one worker client per node, and runs the same closed-loop create-claim-submit cycle. The cluster boots are real — raft leadership election, log replication, FSM apply, the works — but they all happen on one machine over loopback. The 9-10K cycles/s number is what one machine running three processes can sustain end-to-end. `pkg/app/raft_smart_routing_bench_test.go` is the corresponding HTTP-transport bench, kept for the comparison value of seeing what the gRPC migration actually bought.

The variance is real and worth naming. WSL2 specifically can shift bench numbers by 2× run-to-run depending on what else is happening on the Windows host — we have observed a clock jump of more than 50 minutes inside a single bench window on WSL2, with no warning and no log entry. Bare-metal Linux is much more stable but still shows ±10-20% variance between runs on the same hardware. Take the median of three or five runs, not the headline of one. The historical numbers across releases are tracked in `docs/30-performance-baselines.md`; the k6 load-test harness for testing realistic HTTP client workloads (separate from the in-process Go benches above) lives in `docs/26-load-testing.md`. If you are sizing a production deployment, run both: the Go benches tell you what the server can do; k6 tells you what your clients can elicit.

## Reading the numbers honestly

Every number in this chapter came from one bench file on one machine. Three things move them. Hardware: CPU core count, disk fsync latency, NIC round-trip time, NUMA topology. Workload shape: how big are the task payloads, how concurrent are the producers, how many tasks per batch, what is the worker's actual completion latency. Configuration: how many Pebble shards, is `FsyncOnCommit` on, is raft on, is the Apply coalescer doing useful work for your batch size, what does your Pebble cache hit rate look like.

When you measure your own workload, expect a different number than 76,639 tasks/s. That number was produced by `internal/bench/profile_full_cycle_test.go` on a specific 12-core box with NoSync mode and synthetic 64-byte tasks. Your tasks are probably bigger. Your durability requirement probably forbids NoSync. Your client count, batch size, and topology are not the bench's. The numbers in this chapter are reference points, not capacity plans.

Run the bench against your workload before sizing. If you are deciding between single-node and 3-node raft for an HA requirement, run `internal/bench/profile_full_cycle_test.go` on your single-node candidate hardware first to establish the ceiling, then run `pkg/app/raft_grpc_bench_test.go` on three of those boxes to measure your actual cost of HA — not the loopback cost reported here. If the numbers come back close to ours, pattern-match against this chapter. If they come back different — and they probably will — trust your measurements over ours.
