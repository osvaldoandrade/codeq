Simple Loops

Implement workers in 10 lines of code. Just poll, process, and complete.

# 1. Enqueue a task from the terminal
$ codeq task create \
    --event render_video \
    --priority 10 \
    --payload '{"jobId": 500}'

✔ Task created: a5a4d2ad...

# 2. Start a worker for this event type
$ codeq worker start \
    --events render_video \
    --concurrency 5

ℹ Starting worker pool (5 threads)...
✔ Worker connected. Listening...

# 3. Inspect queue depth
$ codeq queue inspect render_video

READY: 124 | DELAYED: 5 | IN_PROGRESS: 5
package main

func workerLoop() {
    client := codeq.NewClient("http://localhost:8080")

    // The infinite loop pattern
    for {
        // 1. Claim (Pull with lease)
        task, err := client.Claim(
            []string{"render_video"}, 60, // leaseSeconds
        )
        
        if task == nil {
            time.Sleep(5 * time.Second) // Backoff
            continue
        }

        // 2. Process Work
        processVideo(task.Payload)

        // 3. Complete (Ack)
        client.Complete(task.ID, Result{Status: "COMPLETED"})
    }
}

Frequently Asked Questions

Deep dive into design decisions and operational models.

Why codeQ and what problem does it address?
codeQ is a reactive task scheduler designed to bridge the gap between simple in-memory queues and complex message brokers. It addresses the need for persistence, delayed delivery, and strict ownership (leases) in distributed systems, without the operational overhead of managing heavy clusters like Kafka or RabbitMQ. It is specifically optimized for agentic workflows where tasks might be long-running or require reliable retries.
Why KVRocks instead of Redis?
While Redis is excellent for speed, it stores everything in RAM. KVRocks implements the Redis protocol but stores data on SSDs (RocksDB). List operations remain O(1), with only a constant latency overhead (RAM vs SSD) compared to Redis. This means you get the same algorithmic efficiency at a fraction of the cost, with full disk persistence.
How does the Lease/Claim model work?
Unlike "fire-and-forget" systems, codeQ requires workers to claim a task, which grants an exclusive lease for a specific duration (e.g., 60s). If the worker crashes or fails to complete the task before the lease expires, codeQ automatically returns the task to the ready queue. This guarantees at-least-once processing and prevents tasks from being lost in limbo.
Is delivery strictly FIFO?
codeQ prioritizes availability and throughput over global ordering. Tasks are FIFO within a specific shard and priority tier, but strict global ordering is not guaranteed across shards. This design decision allows the system to scale horizontally without being bottlenecked by a single locking mechanism.
How does it handle retries and failures?
If a task fails, workers can send a NACK signal. codeQ then calculates a backoff delay (Linear, Exponential, or Jittered) and moves the task to a Delayed Queue. If the number of attempts exceeds the configured limit, the task is moved to a Dead Letter Queue (DLQ) for manual inspection, ensuring bad tasks don't clog the system.
How do Agents authenticate?
codeQ enforces a strict security model. Producers (Agents creating tasks) use Bearer tokens validated against an Identity Provider. Workers use JWTs with specific scopes (e.g., codeq:claim) validated against a JWKS endpoint. This ensures that only authorized entities can access specific queues or event types.
